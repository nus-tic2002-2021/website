    My comments:

    • If the workload is too high: I realize this module can feel heavy, especially if you don't have prior programming experience, and also if you have a day job too.
      That said, SE modules are usually heavy because learning SE skills needs a lot of practice.
      I try to alleviate the workload problem by keeping deadlines fairly flexible and maintaining a low bar for the minimum you need to do to get a decent grade.
    • If the workload is too low:
      • Try to do more in the project: The project is open ended are you are encouraged to do as much as you want. You can refer to Duke full description to find more things you can do with it.
      • Learn the optional topics: I have added a lot more optional topics to the second half of the semester. This means there is more to learn every week, if you have the capacity to do so.

    Some thoughts & feedback:

    • Perhaps we could start learning Design and UML (and various modelling techniques) earlier on (say before recess week) and keep revising throughout the semester.
    • Hope we can cover same topics as CS2103T class. At least the theory part.
    • I think students who got to this module have already done Data Structures & Algorithms, and 2 programming classes in C++. In TIC1002, we briefly studied 4 principles of OOP. While I appreciate the recap on OOP, we could probably go faster since it's not a foreign concept. We also learnt some OOP style programming in TIC1002, implementing ADT's (e.g. stack and queues) in OOP style.
    • Furthermore, in the BTech group, I believe many are already working programmers, with some experience with Java, C# and similar languages. I think what we lack is not practical programming experience, but instead, the theory and rigorous way to approach Software Engineering. e.g. Design, OOP, UML, documentation writing, or any other concepts/ideas that are hard to acquire formally outside of academic setting.
    • I would have preferred more interaction with the rest of class through forums or discussions on Microsoft teams. During our intro programming class (TIC1001), the Coursemology forums were pretty active, so I know there are some people in our batch who love to help others.

    Overall, so far, I will say that I am enjoying this module! Especially the open-ended-ness of the project component, which gives the flexibility to self-direct our learning and take it as deep as we want via the optional exercises. Yet, I think the course can be even more targetted to BTech's working programmers by giving more airtime on the concepts that are hard to pick up on our own, e.g. design principles and explaining a good software design through UML and writing precise documentation. I think this is somewhat already reflected in the grading scheme with high emphasis on documentation (10%) and theory (60%); it would be great to start thinking about these things earlier on at the start of module and throughout the semester.

    • I'll consider frontloading the semester a bit more next time, although I don't want to overwhelm those who are struggling at the start either.

    • I've added almost all CS2103T topics to this module, although some of it are added as optional topics.

    • Sounds like you did well in prior programming modules. See my comment on pace/workload above to learn how you can get more from the module.

    • Second half of the semester cover more theory (after giving inexperienced students some time to catch up), and there are optional topics if you want to go deeper.

    • I've enabled the Coursemology forum after you've brought up this point. I guess it was a bit too late to get students involved in it. Will do so from the start next time.

    • Glad to hear you are enjoying the module. Yes, I'll try to give more things for experienced programmers in the module to do. But those things will have to be optional, as experienced programmers are a minority (about 15% in this cohort) and such experience is not formally a prerequisite of the degree program.


    My general responses for points raised multiple times

    Given below are some general responses to points you guys raised multiple times above:

    • Looking for more programming practice?: I'm afraid I wont be able to add more exercises this time. Will consider adding more optional exercises next semester. In the meantime, you can get more programming practice by doing more in the project. You can refer to Duke full description to find more things you can do with it.
    • Looking for more feedback on code written?: Unlike lower-level programming modules, giving specific feedback on code written in an SE module takes a lot more expertise/effort, and such feedback is still rather subjective. We do not do that in any of our SE modules. Instead, we train students to slowly learn how to self-judge your own code. In addition, we'll be teaching you some code quality guidelines that you can apply to your code later.
